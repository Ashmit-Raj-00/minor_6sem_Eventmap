4. ABSTRACT (Maximum 2 Pages)
EventMap is a web-based event management application that provides a map-centric
interface for discovering and interacting with events. Unlike traditional event
apps that rely only on list pages, EventMap presents events directly on an
interactive map so that users can understand spatial context (distance, nearby
options, and venue placement) before deciding to participate.

The application supports the complete basic event lifecycle: user registration
and login, browsing and searching events, creating and managing events and
sessions (for organizers/admin), joining events (for attendees), and viewing
participants and leaderboards. To encourage engagement, the system implements a
lightweight gamification model (points/XP, level, global leaderboard, and per-
event leaderboard). Points are awarded for meaningful actions such as creating
events, joining, tagging, and check-in.

From a technical perspective, EventMap is implemented as a client–server web
application:
  - Backend: Go (net/http) REST APIs secured using JWT authentication with basic
    Role-Based Access Control (RBAC). Middleware handles authorization, logging,
    request IDs, CORS, and rate limiting.
  - Frontend: Static HTML/CSS/JavaScript UI using Leaflet and OpenStreetMap
    tiles for a modern map experience, plus mobile-friendly navigation and
    filters (tag, radius, search).

The system performs nearby discovery and location gating using geographic
distance computation (Haversine formula). On-ground engagement is enabled via
location-based operations:
  - Tagging is permitted only when the user is near the event location.
  - Check-in is permitted only when the user is inside a configurable radius.

For data persistence, the backend maintains runtime state in memory and saves it
to a local CSV-based storage under the data/ directory. The storage location can
be configured. In addition, background goroutines and buffered job queues
simulate asynchronous notification and analytics pipelines. The solution can be
executed locally for development and demonstration, and it is also deployment-
ready using Docker. The frontend can be hosted as static files (e.g., Netlify),
while the backend can be deployed separately on any platform that runs Docker or
a Go binary.

5. INTRODUCTION
5.1 Purpose
The purpose of EventMap is to provide a simple and intuitive platform for event
management with a focus on location. The system is designed to help users find
relevant events quickly and help organizers manage events without needing a
complex portal.

Primary goals:
1) Improve event discovery using a map-first UI.
2) Provide role-based operations (attendee vs organizer/admin).
3) Provide location-based engagement (tag and check-in) to make participation
   more meaningful and to reduce fake attendance.
4) Provide motivation and interaction through an XP/level and leaderboard model.

5.2 Scope
In-scope:
- Register/Login with JWT
- RBAC roles: attendee, organizer, admin
- Event + session management
- Join, participants tracking
- Nearby events search (Haversine)
- Location-gated tag/check-in
- Points/XP + global/event leaderboards
- CSV persistence (data/)

Out-of-scope (MVP):
- Payments/ticketing
- Real-time chat/push notifications (only simulated)
- Full admin console
- Advanced GIS (routing/geocoding/clustering)

5.3 Features
Frontend:
- Map view with markers and popups
- Search + tag filter + radius filter
- Geolocation “Locate”
- Auth (login/register) + guest browsing
- Create event using pinned map coordinates (organizer/admin)
- Join, tag, check-in, view tags
- Global and event leaderboards

Backend:
- REST APIs: auth, events, sessions, participants, leaderboards
- JWT middleware + RBAC middleware
- Rate limiting, request IDs, logging
- Async analytics/notification queues
- CSV load/save persistence

6. LITERATURE REVIEW (Elaborated)
This project is influenced by and related to multiple areas of software systems:

6.1 Event Management Systems
Typical event systems provide workflows such as event creation, time scheduling,
participant enrollment, and basic discovery. Many existing systems are list-
driven and focus on form-based administration. EventMap implements a minimal
version of these ideas while prioritizing ease-of-use and quick operations.

6.2 Location-Based Services (LBS) and Geo-Fencing
Location-based services use the user's geographic position to provide relevant
content. Common patterns include:
  - Nearby search: listing items within a radius around the user's location.
  - Geo-fencing: restricting an action to users physically present inside a
    boundary (for example, check-ins or access control).
EventMap applies these ideas using distance computation to filter and validate
requests for tagging and check-in.

6.3 Map-First User Interfaces
Modern systems frequently integrate maps for discovery and navigation. A map
reduces cognitive load when events are location-dependent and can help users
decide based on distance and accessibility. EventMap uses Leaflet (client-side
mapping library) with OpenStreetMap tiles (open map data) as a cost-effective
and widely used approach.

6.4 Security and Authorization in Web APIs
Token-based authentication (JWT) is a common approach for stateless
authentication in REST APIs. RBAC is also widely used to restrict privileged
operations (create event, manage sessions, view participants). EventMap follows
these standard design principles through middleware enforcement.

6.5 Gamification for Engagement
Gamification approaches such as points, levels, and leaderboards are widely used
to increase participation and retention. EventMap applies a simple points model
so that core actions translate into measurable progress and community ranking.

7. SYSTEM ANALYSIS

7.1.1 Product Perspective
EventMap is a standalone web application with:
  - A client UI served as static files.
  - A backend HTTP server exposing APIs.

The system is designed so that persistence can later be replaced with a
relational database without major changes to the client experience.

  - May use mobile devices; location permission may be available or denied.
Organizer:
  - Creates and updates events and sessions.
  - Requires clarity on event location (pinned coordinates) and schedule.
Admin:
  - Higher privilege for management and maintenance.

Stakeholders:
- Attendee: browse/search, nearby, join, tag/check-in, leaderboards
- Organizer: attendee features + create events/sessions, view participants
- Admin: organizer privileges + default seeded admin support

FR3 List/search/filter events (tag/radius)
FR4 Organizer/Admin create event (title/times/location/tags/radius)
FR5 Organizer/Admin create sessions under events
FR6 Join event
FR7 Check-in only inside radius
FR8 Tag event only near event location
FR9 Show leaderboards (global + per-event)
FR10 Persist data in CSV and restore on restart

Non-Functional Requirements (NFR)
NFR1 Security: JWT + RBAC for protected routes
NFR2 Performance: fast list/nearby responses for campus scale
NFR3 Reliability: CSV persistence + graceful shutdown
NFR4 Usability: map-first UI for desktop/mobile

  - Sidebar panels for profile/auth, create event, and leaderboards.

8. SYSTEM DESIGN & SPECIFICATIONS
8.1 High Level Design (HLD)
Architecture:
- Browser UI (web/) <-> Go REST APIs (/api/*)
- Storage: CSV files under data/
- Async: goroutines + buffered queues (analytics/notifications)

Module structure (repo):
- cmd/server/main.go
- internal/api (handlers, middleware, JSON helpers)
- internal/auth (JWT sign/verify)
- internal/store (memory + csvdb persistence + models)
- internal/geo (Haversine distance)
- internal/async (background runner)
- web/ (HTML/CSS/JS frontend)

Diagrams:
See report/Minor_Project_Report_EventMap.md and report/diagrams/*.mmd

8.1.1 Project Model (Expanded Component View)
Frontend responsibilities:
  - Render interactive map and markers.
  - Manage auth token in browser storage and attach Authorization header.
  - Capture location (Locate) and show distance to selected event.
  - Submit API calls for create/join/tag/check-in and refresh leaderboards.

Backend responsibilities:
  - Validate input, enforce RBAC, issue and verify JWT.
  - Compute nearby events and distance validation for geo-fenced actions.
  - Persist state changes to CSV on a best-effort basis after writes.
  - Emit analytics/notification jobs to async runner.

Storage responsibilities:
  - Maintain durable state across restarts using CSV snapshots.
  - Support load at server start and save during runtime and shutdown.

8.1.2 Data Storage Design (CSV “Database”)
The system persists data into multiple CSV files (under CSV_DB_DIR, default data/).
Typical files include:
  - users.csv
  - events.csv
  - sessions.csv
  - participants.csv
  - user_points.csv
  - event_points.csv
  - checkins.csv

This design is easy to demonstrate and debug, and it provides persistence
without requiring a database server in the MVP.

8.2 Low Level Design (LLD)
Key Algorithms / Pseudocode
1) Haversine Distance (km)
R = 6371.0
a = sin(dLat/2)^2 + cos(lat1)*cos(lat2)*sin(dLng/2)^2
c = 2 * atan2(sqrt(a), sqrt(1-a))
dist = R * c

2) JWT Authentication (Middleware)
- If Authorization missing: proceed as guest
- Else parse Bearer token, verify signature/exp, load user, attach to context

3) Tag near event (location-gated)
- Normalize tag
- If user farther than 0.75 km from event: forbidden
- Else add tag (if not present) + award points

4) Check-in (geofence)
- Radius = event.checkinRadiusKm (default 0.2 km)
- If farther than radius: forbidden
- If already checked in: conflict
- Else store check-in time + award points

5) Points/XP model
- create event: +50
- join event: +10
- tag near event: +5
- check-in: +30
- level = floor(points/100) + 1

8.2.1 Input Validation Rules (Examples)
Username:
  - 3 to 32 characters
  - Allowed: letters, numbers, underscore, hyphen, dot
Password:
  - 6 to 72 characters
Event time fields:
  - Must follow RFC3339 format at API boundary
Tag normalization:
  - Lowercase, trim, spaces -> hyphen
  - Max length constraint (to keep UI clean)
Check-in radius:
  - Default 0.2 km if not set

8.2.2 Security Design (MVP Level)
Authentication:
  - JWT token issued on login; token is verified on subsequent requests.
Authorization:
  - RBAC denies restricted actions if role is not allowed.
Rate Limiting:
  - Basic per-IP rate limiting to reduce abuse.
CORS:
  - Configurable origin to support separate frontend hosting.

Leaderboards:

9.3 Testing Strategy (Documentation)
Functional testing:
  - Verify register/login and token usage
  - Verify RBAC: organizer can create, attendee cannot create
  - Verify join flow and participant list
  - Verify tagging/check-in denied when far and allowed when near
  - Verify leaderboard changes after actions

API testing:
  - Use Postman or curl to validate endpoints and error codes.
  - Validate rate-limiting response (429) under repeated requests (optional).

Persistence testing:
  - Create users/events, restart server, confirm CSV reload restores data.

9.4 Limitations (MVP)
  - CSV persistence is not ideal for large-scale concurrency and complex queries.
  - Geo-fencing is based on client-reported GPS coordinates (trust model is MVP).
  - No advanced admin dashboard; only API-level controls.

9.5 Future Enhancements
  - Replace CSV store with relational DB (PostgreSQL/MySQL) and add migrations.
  - Add geocoding for addresses and map search.
  - Add clustering for markers and improved map performance.
  - Add real notifications (email/push) and analytics dashboards.
  - Add event moderation and admin user management.
  - Add QR-based check-in to reduce GPS spoofing.
